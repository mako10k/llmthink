# ADR ルール

## 1. 目的

本書は、このプロジェクトで Architecture Decision Record を残す際の作法を定義する。

ADR の目的は、意思決定の正しさを誇示することではない。何を候補にし、どの制約の下で、なぜその判断を採ったかを後から追えるようにすることである。

このプロジェクトでは、ADR もまた「思考監査可能性」を高めるための文書として扱う。

---

## 2. 基本原則

- ADR は結論だけでなく判断過程を残すこと
- 結論が暫定であっても記録すること
- 却下案も残すこと
- 判断時点の制約を省略しないこと
- 後から読んで比較可能な定型フォーマットにすること

---

## 3. ADR を書くべきタイミング

Must:

- DSL の中心概念を追加または変更するとき
- 監査エンジンの責務を追加または削除するとき
- 意味論、MECE検査、契約違反判定の基準を変更するとき
- 永続化形式、実行モデル、監査結果モデルを変更するとき

Should:

- 複数案があり、将来見直しが想定されるとき
- ユーザー体験や記述負荷に影響する制約を追加するとき

書かなくてよい例:

- 明らかな文言修正
- 振る舞いを変えないリファクタリング
- 誤字脱字修正

---

## 4. ステータス

ADR は次のいずれかのステータスを持つ。

- proposed
- accepted
- superseded
- deprecated
- rejected

ルール:

- 初版は proposed または accepted のどちらかで開始する
- 差し替えが発生したら旧ADRは superseded に更新する
- 採用しない案を残す場合は rejected を使う

---

## 5. 必須テンプレート

各 ADR は最低限、次の節を持つ。

1. Title
2. Status
3. Date
4. Context
5. Decision
6. Alternatives Considered
7. Consequences
8. Auditability Notes

### 5.1 各節の意味

- Title: 判断対象を一文で表す
- Status: 現在の採用状態
- Date: 判断日
- Context: 判断時の制約、前提、問題設定
- Decision: 採用する方針
- Alternatives Considered: 比較した案と却下理由
- Consequences: 利点、不利益、今後の制約
- Auditability Notes: 将来どこを見れば見直せるか、どの条件で再判断すべきか

---

## 6. 記述ルール

Must:

- 1つのADRでは1つの判断主題だけを扱うこと
- Decision には採用内容を明確に書くこと
- Alternatives Considered には最低1つの代替案を書くこと
- Consequences には負の影響も書くこと
- Auditability Notes には見直し条件を書くこと

Should:

- Context に判断時点の制約を箇条書きで書くこと
- 曖昧な言い回しより、比較可能な観点を書くこと
- 既存ADRとの関係を明示すること

禁止:

- 結論だけを書いて比較案を省略すること
- 「なんとなく良さそう」だけで判断理由を終えること
- 不利益やトレードオフを無記載にすること

---

## 7. このプロジェクト特有の ADR 観点

このプロジェクトでは、次の観点を優先して評価する。

- 思考記述の明確性が上がるか
- 非論理的思考の抑制に寄与するか
- 監査エンジンの説明可能性を損なわないか
- DSL の記述負荷が過剰にならないか
- MECE、補集合、契約違反の検査と整合するか
- 補助機能が主機能を侵食しないか

---

## 8. 保存ルール

推奨保存場所:

- docs/adr/

推奨ファイル名:

- 0001-brief-title.md
- 0002-another-decision.md

運用ルール:

- 連番は昇順で採番する
- superseded になった ADR も削除しない
- 関連する requirements 変更と同じタイミングで更新する

---

## 9. 最初に作るべき ADR 候補

1. このDSLを思考記述DSLとして定義し、検索DSLを主役にしない
2. エンジンを推論代行ではなく思考監査エンジンとして定義する
3. MECEを真理判定ではなく記述規律として扱う
4. query結果を主機能ではなく補助出力とする
5. 重大度モデルを fatal、error、warning、info、hint とする

---

## 10. ADR テンプレート例

```md
# ADR-0001: エンジンを思考監査エンジンとして定義する

## Status
accepted

## Date
2026-04-30

## Context
- 本プロジェクトの目的は思考の記述明確化である
- 意味論的正確性より記述規律を優先する
- 矛盾、契約違反、MECE不備を検出したい

## Decision
エンジンは推論代行ではなく思考監査エンジンとして実装する。

## Alternatives Considered
- 推論エンジン中心にする
  - 意味評価が主役になり、目的がぶれるため不採用
- 検査機能を持たない単純パーサにする
  - 思考規律の強制力が不足するため不採用

## Consequences
- 監査結果モデルが必要になる
- query は補助出力になる
- 意味近傍は hint 扱いに下がる

## Auditability Notes
- 監査結果カテゴリの追加時に再評価する
- query を主機能に昇格させる提案が出たら再判断する
```
